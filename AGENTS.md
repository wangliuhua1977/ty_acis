## 项目总体目标
本项目只围绕三大核心模块建设：

1. AI智能巡检
2. 派单故障处理
3. 报表统计输出

主线闭环固定为：

**自动巡检发现问题 → 故障进入派单池 → 企业微信群通知处理 → 自动/人工确认恢复 → 报表沉淀结果**

不要把项目扩展成大而全平台。
不要额外发明复杂流程、复杂角色、复杂状态机。

## 技术与实现基线
- 桌面端：.NET 8 Windows 桌面应用
- UI：以当前项目既有 UI 技术栈为准，保持现有分层与目录风格
- 地图：高德地图能力接入思路保持不变

如果当前仓库已有既定分层、命名风格、页面组织方式、服务目录结构，必须沿用，不要随意重构目录。

## 业务实现硬约束

### 1. AI智能巡检模块
必须围绕以下基线实现：
- 巡检组基于接口设备列表自由组合
- 每个巡检组独立配置首次执行时间、每日执行次数、任务间隔
- 同一巡检组串行执行，任务超时则顺延，不并发抢跑
- 支持点位级巡检控制策略
- 支持点位级巡检项目配置
- 在线状态巡检为基础必选项
- 视频播放检查必须有“重试 + 协议切换重试”机制
- 画面异常支持接口返回判定 + 本地截图分析两条来源
- 地图必须实时展示巡检过程
- 点位点击后支持视频预览浮窗
- 故障点位红灯闪烁并与侧边列表联动
- 巡检任务完成后，可按巡检组配置为“上墙复核”或“直接进入派单流程”

### 2. 派单故障处理模块
必须围绕以下基线实现：
- 派单模式按巡检组分别配置：自动派单 / 人工派单
- 故障通知走企业微信群 webhook
- 派单内容必须包含：
  - 当前处理单位
  - 排障维护人 + 11位手机号
  - 上级负责人 + 11位手机号
  - 点位名称
  - 故障类型
  - 巡检发现故障时间
  - 故障截图（如有）
- 同一点位同一故障重复出现时不新建，合并到原记录，并更新最近一次故障时间
- 工单状态保持极简：待派单 / 已派单
- 恢复状态独立：未恢复 / 已恢复
- 恢复后必须自动发送恢复通知
- 管理员允许手工调整当前处理单位、排障维护人、上级负责人及手机号

### 3. 报表模块
首版只围绕：
- 巡检执行报表
- 故障统计报表
- 派单处置报表
- 责任归属报表
- 重点未恢复故障清单

不要扩展复杂 BI、复杂多维分析、复杂权限报表。

## UI与交互硬约束

### 页面结构
一级导航固定为：
- 首页
- AI智能巡检
- 故障派单处理
- 报表中心
- 系统设置

### AI智能巡检页面
必须采用：
- 左侧任务区
- 中间地图中台
- 右侧详情与视频预览区

### 地图要求
- 地图不是装饰，必须是核心工作区
- 列表与地图必须双向联动
- 当前巡检点位高亮
- 故障点位红灯闪烁
- 支持点位点击查看视频预览或状态说明

### 上墙复核页
- 本轮所有巡检点位必须全部上墙
- 每个点位展示截图、巡检结果、简要故障说明
- 支持按正常/故障/单位/故障类型筛选

### 派单处理页
必须采用：
- 左侧筛选区
- 中间工单列表
- 右侧工单详情

## 主题系统约束
系统必须预设 3 套主题方案：
- 电信深海蓝
- 极夜科技蓝紫
- 冰川银蓝暖金

要求：
- 兼顾电信配色与科技风格
- 冷色为主，暖色点睛
- 不得单调
- 不得花哨
- 地图必须跟随主题联动
- 用户可修改预设主题
- 用户可新建自定义主题方案

不要把颜色值散落写死在页面里。
必须通过主题令牌、主题配置或统一资源管理颜色。

---

## 文案与术语系统约束
系统必须支持主要面板文字自定义，包括：
- 导航名称
- 页面标题
- 卡片标题
- 状态文字
- 提示语
- 通知标题
- 报表标题

必须支持行业术语方案切换，例如：
- 电信版
- 通用安防版
- 文旅景区版

必须支持“方案级覆盖 + 单项微调”。
建议通过术语变量机制实现全局联动，不允许大量页面死写固定文案。

---

## 配置约束
### API 接入相关信息
不得在界面中展示以下内容：
- 接口基础地址
- Token
- Key
- 密钥
- webhook 地址
- 其他底层接入参数

统一写入本地配置文件，由系统启动时读取。

界面只呈现业务功能和运行结果，不呈现底层对接细节。

## 开发行为约束

### 必须做到
- 先读基线文档，再动代码
- 尽量做小步迭代，不一次改太大
- 修改前先定位对应模块边界
- 页面字段、按钮、状态命名保持统一

- 不要擅自新增第四个核心业务模块
- 不要把状态机设计复杂化
- 不要把系统做成通用工单平台
- 不要把接口参数做进界面
- 不要把颜色、标题、状态词散落硬编码到各个页面
- 不要无依据重命名已确认模块名
- 不要跳过基线文档直接自由发挥
